DBMS SEMESTER VIVA PITCH - GYM MANAGEMENT SYSTEM

1) 30-Second Project Intro
This project is a role-based Gym Management System built using Flask and MySQL.
It digitizes core gym workflows: member onboarding, trainer assignment, membership plans, subscription lifecycle, and attendance tracking with both check-in and check-out.
I designed it as a practical DBMS application that demonstrates schema design, SQL operations, normalization, and transaction-safe business logic.

2) Problem Statement
Manual gym operations usually suffer from:
- Inconsistent member records
- Hard-to-track active plans and renewals
- No reliable attendance audit
- Weak access control for staff and members
This system solves those problems with a normalized relational model and role-based access.

3) Why This is a DBMS-Focused Project
This is not just a UI app. It is database-driven and demonstrates:
- Proper entity modeling
- PK/FK constraints
- Referential actions (CASCADE, RESTRICT, SET NULL)
- Join-heavy data retrieval
- Aggregation for analytics
- Status-based lifecycle tracking
- Real workflow transactions (member + subscription creation together)

5) Database Schema (Explain in Viva)
Tables:
- User(user_id, username, password_hash, role, member_id, trainer_id)
- Member(member_id, first_name, last_name, email, phone, join_date, trainer_id)
- Trainer(trainer_id, first_name, last_name, specialization, phone)
- Membership_Plan(plan_id, name, duration_months, price)
- Subscription(subscription_id, member_id, plan_id, start_date, end_date, status)
- Attendance(attendance_id, member_id, check_in_date, check_in_time, check_out_time)

Important relationships:
- Trainer 1-to-many Member
- Member 1-to-many Subscription
- Plan 1-to-many Subscription
- Member 1-to-many Attendance
- User maps to role identity (member/trainer/admin)

5A) Detailed Table-Wise Structure (Attributes + Keys)

Table: Trainer
- trainer_id: INT, PK, AUTO_INCREMENT
- first_name: VARCHAR(50), NOT NULL
- last_name: VARCHAR(50), NOT NULL
- specialization: VARCHAR(100), NULL
- phone: VARCHAR(15), NULL

Table: Member
- member_id: INT, PK, AUTO_INCREMENT
- first_name: VARCHAR(50), NOT NULL
- last_name: VARCHAR(50), NOT NULL
- email: VARCHAR(100), UNIQUE, NOT NULL
- phone: VARCHAR(15), NULL
- join_date: DATE, NOT NULL
- trainer_id: INT, FK -> Trainer.trainer_id, NULL

Table: Membership_Plan
- plan_id: INT, PK, AUTO_INCREMENT
- name: VARCHAR(50), NOT NULL
- duration_months: INT, NOT NULL
- price: DECIMAL(10,2), NOT NULL

Table: Subscription
- subscription_id: INT, PK, AUTO_INCREMENT
- member_id: INT, FK -> Member.member_id, NOT NULL
- plan_id: INT, FK -> Membership_Plan.plan_id, NOT NULL
- start_date: DATE, NOT NULL
- end_date: DATE, NOT NULL
- status: ENUM('Active','Expired','Cancelled'), DEFAULT 'Active'

Table: Attendance
- attendance_id: INT, PK, AUTO_INCREMENT
- member_id: INT, FK -> Member.member_id, NOT NULL
- check_in_date: DATE, NOT NULL
- check_in_time: TIME, NOT NULL
- check_out_time: TIME, NULL

Table: User
- user_id: INT, PK, AUTO_INCREMENT
- username: VARCHAR(50), UNIQUE, NOT NULL
- password_hash: VARCHAR(255), NOT NULL
- role: ENUM('admin','member','trainer'), NOT NULL
- member_id: INT, UNIQUE, FK -> Member.member_id, NULL
- trainer_id: INT, UNIQUE, FK -> Trainer.trainer_id, NULL

6) Normalization Justification
- 1NF (First Normal Form):
	- All attributes are atomic (no multi-valued columns like list of plans in one field).
	- Each row is uniquely identified by a primary key.

- 2NF (Second Normal Form):
	- All tables use single-column surrogate primary keys, so partial dependency does not occur.
	- Every non-key attribute fully depends on the whole primary key.

- 3NF (Third Normal Form) - Highest Form Achieved:
	- No transitive dependency among non-key attributes.
	- Example: Plan price and duration are in Membership_Plan, not copied into Member table.
	- Example: Trainer specialization is stored in Trainer table, not duplicated inside Member records.
	- This reduces redundancy and prevents update anomalies.

Highest Normal Form in this DB system: 3NF.
Reason we stop at 3NF: It gives a good balance of consistency and practical query performance for this project size.

7) Key SQL Concepts Used
- JOIN: member + trainer, subscription + member + plan
- LEFT JOIN: preserve unmatched rows for reporting
- Aggregation: total members, active subscriptions, revenue
- ORDER BY + LIMIT: latest attendance and latest active subscription
- Transactional inserts and updates with commit/rollback

8) Critical Workflows You Can Explain
Workflow 1: Add member with plan
- Insert member
- Get member_id
- Insert active subscription using selected plan duration
- Commit or rollback

Workflow 2: Member plan change
- Cancel currently active plan
- Insert new active subscription
- Keep history for auditability

Workflow 3: Attendance check-out
- Check-in inserts open row (check_out_time NULL)
- Check-out updates latest open row
- Prevent invalid check-out when no open session exists

11) Common Viva Questions + Crisp Answers
Q1. Why separate Subscription from Membership_Plan?
A: Plan is master data, subscription is transactional data. This allows one plan to be reused by many members and keeps historical records of member plan changes.

Q2. Why keep cancelled/expired records instead of deleting?
A: For auditability, analytics, and history. Deleting would lose business information.

Q3. How is data integrity maintained?
A: Through PK/FK constraints, role-based route checks, controlled state transitions, and DB transaction rollback on failure.

Q4. Why nullable check_out_time?
A: It models real-time open sessions. Null means currently checked in; non-null means completed attendance.

Q5. What normalization level did you target and why?
A: 3NF. It avoids redundancy and update anomalies while keeping joins manageable for this project scale.
